Formative Feedback

For this deliverable, we were tasked with evaluating group BKC.

Testing

User testing was conducted with 3 test subjects that we deemed fit for the user population described in BKC's description. The tests were done on Android and conducted as described in the BKC test script. The results can be seen here.

Results and Analysis

Test Plan Critique

User Manual

The user manual provided was very thorough and was useful for testing. However, one issue that we came across was in the method for creating an Android emulator, there were no instructions to then allow that emulated device to access the host WiFi. This was a necessary step so it would have been useful to have documentation. The prototype also did not function on the enumlated Android device, despite following the steps laid out in the manual.


Usability Goals

Comments for the usability goals are under each provided goal in bold.

Achieve goals in a timely manner

Rationale for goal: During our prototype testing, users informed us that the core functionality they wanted from their phones was to access the application they wanted as quickly as possible.

How we'll measure meeting this goal: In order to gauge the completion of this goal, examiners will time how long it takes for test subjects to go from the command prompt to the application they have been directed to access.

We believe this goal was a useful one to include because it gives a clear idea of the time taken, however, it didn't take into account the differing use speeds that users may have and the fact that some users lack familiarity with an android operating system.


Achieve goals with little to no resistance (number of tasks)

Rationale for goal: One of our principal motivations for the design of our application was that users should be less distracted when trying to achieve a particular goal.

How we'll measure meeting this goal: Evaluators will count the number of separate tasks

This goal was a litte vauge to be able to evaluate, especially considering that the numbers of tasks was provided in the testing plan. This could be more easy measured in an open ended test or an observation of a user where set tasks are not provided.


Transitions from one activity to another should be seamless (time)

Rationale for goal: One of our principal motivations for the design of our application was that users should be less distracted when trying to achieve a particular goal.

How we'll measure meeting this goal: Evaluators will measure the time it takes to go from one task to another.

This goal is a repeat of the first goal, so does not evaluate any new charateristics of the system.


Test Procedure and Reporting

The test procedure was easy to understand, however, we were unsure as to what exactly the timer was needed for because it was not mentioned elsewhere. It would have been useful to have been provided with a structed document that we could fill in for observations, instead of the free form document that was provided.

We were also expecting to be provided with material such as a pre and post-test questionnaire and a formal user intruction. This meant that we were unsure of what exactly the user was supposed to know about the application before beginning testing. This was also difficult because a short description of the project could only be found in the project proposal, and we were unsure in many manjor changes had been made since then so could not use it as a description.

Usability Evaluation

The usability evaluation section actually included a test script which was useful but should have been included in the reporting section. There was also a note at the top of the instructions that said all items in bold are mandatory and must be asked, however, only one line was in bold, which lead to some confusion about what was mandatory and what was bonus information.

The section was missing any mention of specific heuristics from Nielsen's list. It also did not have any justificiation for the selected testing goals.


Design Critique

Through our testing, we were able to discern several areas with room for improvement. These suggestions are outlined below:

1. The Design

Something all three of our testers commented on was the user interface of the program. There was initially confusion over the background, as the testers believed that is was part of the program, rather than the background of the actual phone. After that was clarified for them, they suggested several improvements. One suggestion was that the analog clock displayed be changed to a digital one. Our team agrees with this suggestion, as a lot of younger people now are growing up with purely digital clocks, so this alteration may make the program accessible to a wider audience.

Another suggestion was to make the writing on the home page larger, or make it a different colour. One of our testers wore glasses, but found it difficult to see and understand where she should be writing on the home page.

2. Help Documentation

When our testers were asked whether they felt they had support using the system, they all answered not at all. They explained this by saying that it was unclear what they were meant to be doing on the home page, and if they had run into issues, there was no documentation they could turn to. For example, one of our testers noticed that, if they entered a space after the name of an app they were looking for, the search function wouldn't work. The space had to be removed in order for them to continue. Documentation specify how to phrase a search could have been helpful in this situation. Our team thinks it could be very beneficial to include instructions and an area where users could refer to if there is a problem while they use the app.

3. App Accessability

Two of our testers pointed out that having to manually write in the name of the application they wanted to access would be difficult the more apps they had on their phone. Unless they had very good memories, they would just forget what other programs were available to them. When looking through the User Manual, we saw that there is a function that allows seeing the active applications, however, no testers noticed this feature. Our team suggests making this possibility much more visible. Another option we came up with was that possibly applications could be grouped by category. For example, when a user wants to watch movies, they can just type in, "Movie," and different applications will come up (Netflix, Youtube, Amazon Prime Videos etc.)

4. Fluidity

Several of the testers said that due to the tedious task of entering what app they want to access each time, they probably wouldn't use the product for very long. A possible improvememt would be to use something similar to tabs on a Chrome browser, so it is easier and more fluid to navigate between applications.
